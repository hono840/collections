# carskiida 詳細機能仕様（確定版 / feature-planner）

> 作成: 2026-06-07 / 担当: feature-planner（CPO委譲）
> ステータス: 確定版（Hiro承認方針＝二層構造・source属性・輸入車含む・趣味/ポートフォリオ を反映）
> 前提資料: `carskiida-prd-draft.md`

## 0. このドキュメントの読み方

「広さ（breadth）= vPIC/Wikidata 自動収録」と「深さ（depth）= 厳選ショーケース車種の手厚い作り込み」の二層構造を、機能・データモデル・受入条件のレベルまで一貫して落とし込んだ確定仕様。

| 用語 | 定義 |
| --- | --- |
| **breadth 車種 / 浅い車種** | vPIC・Wikidata から自動収録された母集団。最小スキーマのみ保持。`depth_level = "breadth"`。 |
| **depth 車種 / ショーケース車種** | 人手で世代史・パーツ構造・工場まで作り込んだ厳選車種。拡張スキーマを持つ。`depth_level = "showcase"`。 |
| **source（出典）** | 全データフィールドに付与する出典メタデータ。`{ type, label, url, retrieved_at, confidence }`。 |
| **諸元（spec）** | 寸法・重量・エンジン・性能・燃費などの数値仕様。 |
| **世代（generation）** | 同一車名のフルモデルチェンジ単位。例: トヨタ86 ZN6 / GR86 ZN8。 |
| **グレード（grade）** | 世代内の装備/パワートレイン違い。例: GR86 RZ / SZ。 |

## 1. プロダクトスコープ確定（広さ/深さの境界線）

### 1.1 二層構造の機能的な切り分け

| 観点 | breadth（広さ） | depth（深さ / ショーケース） |
| --- | --- | --- |
| 収録手段 | vPIC + Wikidata の自動ETL | 人手キュレーション（事実値転記＋ナラティブ執筆） |
| データ粒度 | メーカー / 車名 / 年式レンジ / ボディタイプ / 原産国（最小） | 世代史・グレード別諸元・パーツ構造・工場・世代ナラティブ |
| 詳細ページ | 「ライト詳細」テンプレ（要約＋出典＋"深掘り募集"表示） | 「フル詳細」テンプレ（世代タブ・パーツアコーディオン・工場・タイムライン） |
| 検索/一覧 | 全件対象 | 全件対象＋「ショーケースのみ」フィルタで絞り込み可 |
| 比較表 | 比較対象に選べる（揃わない項目は欠損表示） | 比較対象に選べる（項目が揃いやすい） |
| source | 自動付与（vPIC/Wikidata） | フィールド単位で手動付与（諸元はメーカー諸元など） |

**設計原則**: breadth と depth は別テーブルではなく、同一の `car_model` を基底に「depth拡張テーブル群がぶら下がるかどうか」で表現する。これにより検索/比較は両者を透過的に扱える。

### 1.2 MVP（P0）スコープの確定

MVPの合格ライン = **「広さの母集団が検索でき、深さのショーケースが図鑑として成立し、両者を比較表で横断できる」**。

MVPに含む（P0）:
1. breadth 自動収録（vPIC + Wikidata ETL バッチ。初期 数百〜数千モデル規模、品質優先で件数は可変）
2. depth ショーケース車種（MVP着地目標: **8〜12 車種 × 各2〜3世代**）
3. 車種詳細（depth=フル詳細 / breadth=ライト詳細 の2テンプレ）
4. 検索・一覧（メーカー別 / ボディタイプ別 / キーワード / ショーケースのみ トグル）
5. 比較表（2〜4台、行=項目・列=対象、差分ハイライト、欠損項目の明示）
6. 用語インライン解説（ツールチップ、用語辞書）
7. source 表示（全フィールドに出典バッジ、詳細ページに出典一覧）

MVPで削る（P1以降）:
- 関連車回遊レコメンド（P1）、お気に入りコレクション（P1）
- 工場の地図ビジュアル（P2）、UGC投稿（P2）、ゲーミフィケーション（P2）
- 高度な数値レンジ・多軸ファセット検索（P1）、ネイティブアプリ（対象外）

恒久スコープ外: 在庫・価格・購入導線。

### 1.3 depth ショーケース車種の選定基準（MVP）

優先度順に評価し、合計 8〜12 車種を確定:

1. **世代史が語れる**: 3世代以上の歴史。例: スカイライン、シビック、ロードスター。
2. **生産地の物語がある**: 工場が変遷/複数工場。例: GT-R（栃木・匠組立）、シビック（鈴鹿/海外）。
3. **パーツ構造の解説価値**: 駆動方式・サス形式・エンジンレイアウトに特徴。
4. **データ入手性**: メーカー諸元・Wikipedia・Wikidata が揃う。
5. **輸入車を最低1〜2車種含む**。例: ゴルフ、ミニ、911 等。
6. **ボディタイプの多様性**: スポーツ/セダン/SUV/コンパクト/ミニバン を分散。

MVPは「国産6〜8 + 輸入2〜4」を前提に受入条件を設計。

## 3. コア機能 詳細仕様

### 3.1 [P0] 車種詳細（depth=フル詳細 / breadth=ライト詳細）

テンプレ分岐は `depth_level` で行う。

**フル詳細 / depth**: ヘッダー（メーカー/車名/ボディタイプ/通称/原産国/ショーケースバッジ）、世代切替（タブ or 水平タイムライン）、諸元セクション（グレード別、各値に source バッジ）、パーツ構造（カテゴリ別アコーディオン）、生産地（世代/グレードごとの工場・原産国）、世代ナラティブ（Markdown）、出典一覧。

**ライト詳細 / breadth**: ヘッダー（メーカー/車名/年式レンジ/ボディタイプ/原産国）、要約、取得済み諸元（欠損は「データなし」明示）、深掘り募集表示、出典（vPIC/Wikidata）。

入出力: ルート `/cars/[modelId]`（任意で `?gen=` / `?grade=`）。SSR/SSG前提。
振る舞い: `showcase`→フル詳細、`breadth`→ライト詳細。不正な `modelId`→404。不正な `?gen`/`?grade`→デフォルトにフォールバック（500にしない）。

### 3.2 [P0] 検索・絞り込み・一覧

フィルタバー（メーカー複数選択/ボディタイプ/原産国/ショーケースのみトグル/キーワード）、カードグリッド、ソート（関連度/車名/メーカー）、件数表示、ページネーション、各カードから「比較に追加」（最大4）。
入出力: `?q=&maker=&body=&country=&showcaseOnly=&sort=&page=`。
振る舞い: キーワードは車名・通称・メーカー・別名に部分一致。フィルタはAND（メーカー内OR）。0件は空状態。比較5台目で警告。

### 3.3 [P0] 比較表

列=対象車（最大4）、行グループ（寸法/重量/パワートレイン/性能/燃費/駆動/生産地/depthならパーツ）。差分ハイライト（数値=最大/最小/同値、文字=不一致）。欠損は「—（データなし）」で差分計算から除外。各セルに出典バッジ。列追加/削除/並べ替え、共有用URL。
入出力: `/compare?ids=modelA:genA:gradeA,...`（最大4）。
振る舞い: 世代/グレード単位で指定可。単位は項目ごと統一（内部正規化値、表示は併記 例:152kW(207PS)）。1台以下→比較不可表示。

### 3.4 [P0] 用語インライン解説
対象用語に点線下線、ホバー(PC)/タップ(モバイル)でポップオーバー。`glossary_term`: `{ slug, term, reading, short_def, long_def?, source }`。詳細/比較/一覧で共通利用。未登録はプレーン表示。

### 3.5 [P0] source（出典）表示
フィールド/セルに出典バッジ（vPIC/Wikidata/メーカー/Wikipedia/UGC）。ホバー/タップで `{label,url,retrieved_at,confidence}`。詳細ページ末尾に出典一覧。`confidence=low` は「要確認」マイクロラベル。

### 3.6 P1/P2
関連車回遊(P1, 価格は扱わない)、お気に入り(P1, 未ログインlocalStorage)、工場地図(P2)、UGC(P2)、ゲーミフィケーション(P2)。

## 4. 受入条件（Given-When-Then）

### 4.1 車種詳細（depth フル詳細）
- AC-D-01: depth車種詳細を開く→世代切替・諸元・パーツ構造・生産地・世代ナラティブ・出典一覧が表示。
- AC-D-02: 別世代タブを選択→当該世代に切替、URLが `?gen=` で更新。
- AC-D-03: 諸元値→各値に source バッジ。
- AC-D-04: パーツカテゴリのアコーディオンを開く→当該パーツ詳細と出典。
- AC-D-05: 生産地→世代/グレードごとの工場名・原産国が出典付き。
- AC-D-06: 不正な `?gen=` →500を出さず最新世代にフォールバック。

### 4.2 車種詳細（breadth ライト詳細）
- AC-B-01: breadth車種→ライト詳細テンプレ。世代タブ/パーツアコーディオンは非表示。
- AC-B-02: 諸元欠損→「データなし」明示。空欄や0でごまかさない。
- AC-B-03: 存在しない modelId→404。

### 4.3 検索・絞り込み・一覧
- AC-S-01: メーカー選択→当該メーカーのみ、件数更新。
- AC-S-02: ボディタイプ+メーカー→AND結果。
- AC-S-03: キーワード→車名/通称/別名一致。
- AC-S-04: ショーケースのみON→showcase のみ。
- AC-S-05: 0件→空状態+解除導線（エラーにしない）。
- AC-S-06: 「比較に追加」→トレイ件数増。
- AC-S-07: 4台ある状態で5台目→拒否+警告。
- AC-S-08: breadth/depth混在→両方表示、depthにショーケースバッジ。

### 4.4 比較表
- AC-C-01: 3台→行=項目・列=3対象。
- AC-C-02: 数値項目→最大/最小がハイライト。
- AC-C-03: 一部欠損→「—」で差分計算から除外（最小値誤判定しない）。
- AC-C-04: breadth/depth混在比較→同一表、揃わない項目は欠損表示。
- AC-C-05: 1台のみ→「2台以上を選択してください」。
- AC-C-06: 出典バッジホバー→source詳細。
- AC-C-07: 単位の異なる項目→統一単位（併記）で比較可能。
- AC-C-08: 世代/グレード指定で比較→指定値が列に反映。

### 4.5 用語インライン解説
- AC-G-01: 登録用語にホバー/タップ→解説ポップオーバー。
- AC-G-02: 未登録用語→リンク化されずプレーン表示。
- AC-G-03: 3画面で同一解説が共通利用。

### 4.6 source 表示
- AC-SRC-01: 任意フィールド→出典バッジ付与。
- AC-SRC-02: url持つsource→新規タブ（`rel="noopener nofollow"`）。
- AC-SRC-03: confidence=low→「要確認」マイクロラベル。
- AC-SRC-04: depth詳細末尾→全出典一覧。
- AC-SRC-05: source未確定→「出典未確認」扱いで確定済みと区別。

### 4.7 横断（非機能・データ品質）
- AC-X-01: breadth ETL後→各モデルに source と depth_level=breadth が付与。
- AC-X-02: depth公開→各フィールドに source 手動付与（source無しは公開不可）。
- AC-X-03: モバイル幅375px→レイアウト崩れなく主要操作可能。
- AC-X-04: 詳細ページ→車名・メーカー・世代を含むSEOメタ。

## 6. データモデル確定提案（二層構造 × source属性）

### 6.1 設計の核
- breadth/depth は同一 `car_model` を共有。`car_model.depth_level` が単一の真実。
- source は埋め込み（embedded）方式: 各エンティティに `field_sources: Record<fieldName, Source>`（JSON）。MVPはJSON埋め込み開始、件数増で関連テーブルへ移行可能。
- 数値諸元は SI正規化値（比較用）と表示用併記文字列を別に持つ。

### 6.2 source オブジェクト
```
Source {
  type: "vpic" | "wikidata" | "wikipedia" | "manufacturer" | "ugc" | "editor"
  label: string
  url?: string
  retrieved_at: string   // ISO8601
  confidence: "high" | "medium" | "low"
  note?: string
}
```
値の無いフィールドには source も無い＝「出典未確認」。

### 6.3 breadth用 最小スキーマ
```
Manufacturer { id, name_ja, name_en, country, logo_url?, field_sources }
CarModel {
  id, manufacturer_id, name_ja, name_en, aliases: string[],
  body_type, origin_country, year_range:{from,to?}, summary_ja?, thumbnail_url?,
  depth_level: "breadth" | "showcase",   // ★単一の真実
  search_text, external_ids:{vpic?,wikidata?,wikipedia?}, field_sources, created_at, updated_at
}
```

### 6.4 depth用 拡張スキーマ（showcase のみ）
```
Generation { id, car_model_id, code?, name_ja?, name_en?, year_range, narrative_md?, is_curated, field_sources, sort_order }
Grade { id, generation_id, name, trim_level?, field_sources }
SpecValue { id, grade_id?|generation_id?, spec_key, value_normalized, value_display, unit, source, confidence }
PartCategory { id, key, name_ja, sort_order }
Part { id, category_id, name_ja, name_en?, spec_summary?, detail_md?, field_sources }
GradePart { grade_id, part_id, source }
Plant { id, name, country, region?, lat?, lng?, field_sources }
ProductionAssignment { id, generation_id?|grade_id?, plant_id, year_range?, source }
GlossaryTerm { slug, term, reading, short_def, long_def?, source }
```

### 6.5 breadth ↔ depth の関係
検索/一覧/比較は `CarModel` を基底に動く。depth化フロー: `depth_level` を showcase に昇格→Generation 以下作成→各フィールドに source 手動付与（source無しは公開バリデーションで弾く / AC-X-02）。比較表は `SpecValue.spec_key` を行キーに揃え、`value_normalized` で差分、`value_display` で表示、欠損除外。

### 6.6 インデックス/検索
`CarModel.search_text` に全文/部分一致（pg_trgm or tsvector）。フィルタ列に複合インデックス。比較は `SpecValue(spec_key, grade_id/generation_id)`。

## 7. CPO確認事項（推測で進めず確認を求める）
1. depthショーケースの確定リスト（8〜12車種、輸入車の具体採用）。
2. breadth初期件数の目標（品質優先・数百規模を仮置き）。
3. 画像方針の最終確認（権利確認済みのみ / breadthは原則画像なし）。
4. アカウント要否（MVPはログイン無し）。
5. source埋め込み(JSON) vs 関連テーブル（MVPはJSON開始）。
6. 矛盾値の優先順位（E-07）の確定。

## 8. MVP受入の合格判定（サマリ）
1. breadth ETL で母集団が検索ヒット。
2. depthショーケース 8〜12車種が図鑑として成立。
3. breadth/depth横断で比較表が差分・欠損を正しく扱う。
4. 全フィールドに source 付与・表示、未確認は区別。
5. 用語インライン解説が3画面で共通動作。
6. モバイルファーストで主要導線が破綻しない。
